# Rust移植計画書 — Dash/Plotly版から高性能ネイティブ波形ビューアへ

---

## 1. 目的

`proto_3_1b` で一旦完了した波形ビューア機能を仕様リファレンスとして固定し、
Dash/Plotly の性能限界を回避するため、Rustベースのデスクトップアプリとして再構築する。

主目的は「言語をRustに置き換えること」ではなく、描画・データ保持・ズーム処理の設計を
大容量波形データ向けに作り直すことである。

---

## 2. 背景と課題

### 現行版の到達点

`proto_3_1b` は以下を実現済み。

- Parquet 読み込み
- 1行複数チャンネル重ね表示
- 行追加・削除、Y軸範囲指定
- line / step 表示切替
- チャンネル別の色・線幅設定
- ホバー同期、ズーム・パン同期
- Cursor A/B 計測
- 区間統計
- FFT
- 作図設定 `.pyc.json` 保存・ロード
- 大容量データ向け Min-Max envelope

### 現行版の限界

Dash/Plotly は高機能だが、以下が大容量データではボトルネックになる。

- Pythonサーバーとブラウザ間のデータ受け渡しが重い
- Plotly figure のJSON化・DOM/WebGL反映コストが大きい
- ズーム・パンのたびにDash callbackが連鎖する
- ブラウザ側メモリ消費が増えやすい
- 90M点級データでは「描画点数を抑えても」UI応答性の限界が出る

---

## 3. 技術方針

### 採用方針

| 項目 | 方針 |
|------|------|
| 言語 | Rust |
| アプリ形態 | ネイティブデスクトップアプリ |
| UI | Rust GUI（egui/eframe 等）を第一候補 |
| 描画 | GPUベースのカスタム波形レンダラを目標 |
| データ形式 | 現行と同じ Parquet を継続 |
| 作図設定 | `.pyc.json` 相当の設定ファイルを継続、必要に応じてv2化 |
| 前処理 | 初期段階ではPython版 converter の併用を許容 |

### 非採用方針

Tauri + Plotly.js のように、Rustバックエンド + Web/Plotly描画にする構成は第一候補にしない。
この構成は配布性やバックエンド性能は改善するが、現行の主ボトルネックである
ブラウザ/Plotly描画パイプラインを残すため、根本的な性能改善になりにくい。

## 4. 基本アーキテクチャ

```
Parquet file
  │
  ▼
Data Loader
  - time列検証
  - channel列検出
  - 型変換
  - メタ情報抽出
  │
  ▼
Waveform Store
  - full resolution data
  - channel metadata
  - time index
  │
  ▼
LOD / Tile Cache
  - min/max pyramid
  - 表示範囲ごとの軽量抽出
  - step信号向け保持ルール
  │
  ▼
Renderer
  - 表示中のピクセル幅に応じた描画点だけ生成
  - GPU描画
  - pan/zoom中も全点をUIへ渡さない
  │
  ▼
Analysis Layer
  - cursor値
  - 区間統計
  - FFT
  - 解析は常に生データを参照
```

重要原則:

- 描画は表示ピクセル数に比例させる
- 解析は必ず生データを参照する
- UIイベントは軽く保ち、重い処理は非同期・キャンセル可能にする
- データ読み込み、LOD構築、描画、解析の責務を分離する

## 5. 移植ステップ

### Phase 0: `proto_3_1b` の仕様固定

状態: 完了

目的: Rust版の比較対象を固定する。

詳細計画: [rust_migration_phase0_plan.md](rust_migration_phase0_plan.md)

作業:

- `proto_3_1b` を機能完了版として扱う
- 現行機能一覧を確定する
- 代表データセットを用意する
  - 小: 100k点
  - 中: 数百万点
  - 大: 90M点級
- 作図設定ファイルの保存項目を整理する
- 現行UIの操作仕様をスクリーンショットまたはドキュメントで記録する

成果物:

- 機能チェックリスト
- 代表Parquetデータ
- 作図設定サンプル
- 性能基準の初期値

完了条件:

- Rust版で再現すべき機能と、後回しにする機能が明確になっている

---

### Phase 1: Rust縦切りプロトタイプ

目的: 技術リスクが最も高い「大容量データ読み込み + 描画 + ズーム」を先に検証する。

最小機能:

- Parquetファイルを開く
- time列とチャンネル列を認識する
- 1行1チャンネルを表示する
- pan / zoom ができる
- 表示範囲に応じて min/max envelope を描画する
- 90M点級データでフリーズしない

この段階で実装しないもの:

- 複数行UI
- 作図設定保存
- Cursor
- FFT
- 色・線幅などの詳細設定

重点検証:

- Parquet読み込み時間
- メモリ使用量
- 初回表示時間
- pan / zoom の応答性
- 連続ズーム時の安定性
- OS別の起動・描画安定性

完了条件:

- 大容量データで「表示・パン・ズーム」が現行Dash版より明確に軽い
- 操作中に全点を描画パイプラインへ流していないことを確認できる

---

### Phase 2: 波形ビューア基本機能の移植

目的: `proto_3_1b` のビューアとしての主要操作をRust版に移す。

移植対象:

- 1行複数チャンネル重ね表示
- 行追加・削除
- line / step 表示切替
- チャンネル別の色・線幅
- Y軸範囲指定
- X軸ズーム・パン同期
- ホバー位置同期
- チャンネル値表示
- ライト/ダークテーマ

設計上の注意:

- 複数行でも、各行が独立して全データを再抽出しない
- チャンネルごとのLODキャッシュを共有する
- step表示はピーク保持だけでなくエッジ保持を優先する
- UI状態とデータ状態を分離する

完了条件:

- 日常的な波形確認作業がRust版だけで実施できる
- `proto_3_1b` の基本表示機能と同等以上の操作性がある

---

### Phase 3: 解析機能の移植

目的: 可視化だけでなく、実務で使っている計測・解析機能を移す。

移植対象:

- Cursor A/B
- Δt と周波数換算
- カーソル位置のチャンネル値
- 区間統計
  - Mean
  - Max / Min
  - P-P
  - RMS
  - Slope
- FFT
  - 対象チャンネル選択
  - 窓関数
  - Amplitude / dB
  - 周波数範囲指定

設計上の注意:

- 解析は描画用LODではなく生データを使う
- 大きな区間の統計・FFTはUIスレッドで実行しない
- 計算中キャンセルまたは再実行上書きを可能にする
- FFT対象区間が巨大な場合は警告または上限を設ける

完了条件:

- `proto_3_1b` のCursor/統計/FFTをRust版で代替できる
- 解析中もUIが固まらない

---

### Phase 4: 作図設定・運用機能の移植

目的: 実務運用で必要な保存・復元・配布機能を整える。

移植対象:

- 作図設定保存
- 作図設定ロード
- データパス復元
- 行構成復元
- チャンネルスタイル復元
- テーマ復元
- スクロール/ズーム設定復元
- exe / app 配布

設定ファイル方針:

- 現行 `.pyc.json` は読み込み互換を持たせる
- Rust版固有の項目が必要な場合は `version: 2` とする
- 可能なら拡張子は将来的に `.panely.json` へ移行する

完了条件:

- 現行版で保存した作図設定をRust版で読み込める
- Rust版で保存した設定を再ロードして同じ表示状態に戻せる
- Python環境なしでアプリを起動できる

## 6. 性能目標

初期目標値:

| 項目 | 目標 |
|------|------|
| 100k点データ | 即時表示に近い操作感 |
| 数百万点データ | 初回表示後、pan/zoom が滑らか |
| 90M点級データ | フリーズせず、連続ズームに耐える |
| 描画点数 | 表示ピクセル幅オーダーに制限 |
| 解析参照 | 常に生データ |
| メモリ | Dash/Plotly版より大幅削減 |

測定項目:

- 起動時間
- ファイル読み込み時間
- LOD構築時間
- 初回描画時間
- pan / zoom レイテンシ
- 最大メモリ使用量
- 連続操作時のメモリ増加

## 7. リスクと対策

| リスク | 内容 | 対策 |
|--------|------|------|
| GUI選定ミス | UIフレームワークが業務要件に合わない | Phase 1で描画・操作性を最優先検証 |
| 描画実装の複雑化 | GPU描画やLOD管理が重くなる | 最初は1ch縦切りで検証し、抽象化を後回しにする |
| MAT/CSV変換の再実装負荷 | MATLAB形式対応は細部が多い | 初期はPython converterを継続利用 |
| 作図設定互換 | `.pyc.json` の仕様が暗黙的 | Phase 0で設定項目を固定する |
| 解析結果差異 | FFT/統計がPython版とずれる | 代表データでPython版との数値比較を行う |
| 配布差異 | Windows/macOSで挙動が変わる | 早い段階で対象OSを決め、CI/手動確認手順を作る |

## 9. 判断基準

Phase 1 の結果で継続判断する。

Rust移植を継続する条件:

- 大容量データでDash/Plotly版より明確に軽い
- 連続ズームでUIが固まらない
- メモリ使用量が許容範囲に収まる
- 実装の複雑度が今後の保守に耐える

方針を見直す条件:

- GUI/描画ライブラリの制約で必要な操作性を出せない
- Rust版の実装コストが想定より大きすぎる
- C++/Qt等の方が明確に短期実用化しやすいと判明する

## 10. 最初の着手タスク

最初にやるべきことは、完成版アプリを作り始めることではない。
以下の最小プロトタイプで、性能改善の核心だけを確認する。

```
Rust app
  ├─ Parquetを読む
  ├─ チャンネルを1つ選ぶ
  ├─ time vs value を表示する
  ├─ pan/zoomする
  └─ 表示範囲に応じてmin/max envelopeを使う
```

このプロトタイプが成立すれば、`proto_3_1b` の機能を段階的に移植する価値がある。
成立しなければ、早い段階でC++/Qtなど別案に切り替える。
