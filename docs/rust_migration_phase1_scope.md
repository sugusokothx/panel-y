# Rust移植 Phase 1 範囲定義 — 縦切りプロトタイプ

---

## 1. 目的

Phase 1では、Rust移植の最大リスクである「大容量Parquetを読み、波形を軽く表示し、pan/zoomできるか」を検証する。

この段階では完成アプリを作らない。`proto_3_1b` の全機能を追わず、性能改善の核心だけを縦切りで確認する。

---

## 2. 実装する機能

| ID | 機能 | 必須 |
|----|------|------|
| P1-01 | Rustデスクトップアプリとして起動する | 必須 |
| P1-02 | Parquetファイルを指定して読み込む | 必須 |
| P1-03 | `time` 列を認識する | 必須 |
| P1-04 | `time` 以外をチャンネルとして認識する | 必須 |
| P1-05 | チャンネル一覧を表示する | 必須 |
| P1-06 | 1チャンネルを選択する | 必須 |
| P1-07 | 1行の波形を表示する | 必須 |
| P1-08 | X軸pan/zoomを行う | 必須 |
| P1-09 | 表示範囲に応じたmin/max envelopeを使う | 必須 |
| P1-10 | 90M点級データでフリーズしない | 必須 |
| P1-11 | 基本的な読み込み時間・メモリを記録する | 必須 |

## 4. 技術検証ポイント

### Parquet読み込み

確認すること:

- Rustで対象Parquetを読める
- time列とチャンネル列を取得できる
- 必要な列だけを選択読み込みできるか
- 全列読み込み時のメモリ使用量

### データ保持

確認すること:

- time配列と値配列を効率よく保持できる
- float32/float64の方針を比較できる
- 大容量データでコピーが増えない

### LOD / min-max

確認すること:

- 表示幅に応じて点数を制限できる
- min/max envelopeでピークを落とさない
- pan/zoom時に必要範囲だけ処理できる
- LOD/tile cacheの構築時間とメモリが許容範囲か

### 描画

確認すること:

- pan/zoom中に全点を描画へ渡さない
- 描画点数が画面幅オーダーに収まる
- 連続ズームで固まらない
- GPU描画またはそれに準じる軽さを得られる

---

## 5. 合格条件

Phase 1は以下を満たしたら合格。

- `test_100k.parquet` を読み込んで1ch表示できる
- `panely_large_10s_1mhz_9ch.parquet` 相当の大データを読み込んでもアプリが落ちない
- 大データでpan/zoomが操作不能にならない
- 表示処理が全点数ではなく表示幅に概ね比例している
- 連続ズームでメモリが増え続けない
- Dash/Plotly版より大データの操作感が明確に軽い

---

## 6. 失敗条件

以下の場合はPhase 1の技術方針を見直す。

- GUI/描画ライブラリの制約でpan/zoomが実装困難
- 大データでRust版も全点処理に近くなってしまう
- LOD/tile cacheの実装コストが過大
- Parquet読み込みのライブラリ制約が大きい
- C++/Qtなど他案の方が明確に短期実用化しやすい

---

## 7. 推奨実装順

1. Rustプロジェクトを作る
2. 空ウィンドウを表示する
3. Parquetファイルを固定パスで読み込む
4. time列と1チャンネルだけをメモリに載せる
5. 全体範囲をmin/max envelopeで表示する
6. X軸pan/zoomを追加する
7. 表示範囲に応じて再抽出する
8. チャンネル選択UIを追加する
9. 大データで測定する
10. 結果をPhase 1評価として記録する
