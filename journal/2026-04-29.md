## 判断・設計メモ

- Rust移植では、`proto_3_1b` のUI/解析仕様を正とする
- `proto_3_1` には multiconverter exe化対応があるが、ビューア本体の仕様固定では `proto_3_1b` を優先する
- Phase 1では完成アプリを作らず、以下の縦切りだけを実装する方針
  - Rustアプリ起動
  - Parquet読み込み
  - time列・チャンネル列認識
  - 1ch選択
  - 1行波形表示
  - pan/zoom
  - min/max envelope
  - 90M点級データでフリーズしないことの確認
- 解析機能は描画用LODではなく、常に生データを参照する方針を維持する
- 作図設定は現行 `.pyc.json` の読み込み互換を持たせる
- 小データではDash/Plotly版も十分軽く、Rust版の優位性は中・大データで確認する必要がある
- backend測定では中・大データもズーム再描画自体は軽い
  - Rust版で比較すべき主な差分は、ブラウザ描画・Dash通信・メモリ保持・操作時の一貫性になる
- 作図設定ロード処理自体は小データで約0.63 sだったが、リロード直後のクリックが自動操作上不安定だった
  - これは「ロード処理が重い」と断定せず、「Dash初期化直後の操作受付が不安定な可能性」として分けて扱う
  - Rust版では、ボタンが見えている状態と実際に操作可能な状態を一致させる
- Rust版コード置き場は `proto_4` ではなく `code/rust_phase1` にした
  - Python/Dash版の機能番号とは別系統の「移植Phase」として扱うため
  - `proto_3_1b` を参照実装として固定し、Rust版は技術検証用の縦切りとして進める
- Phase 1の最初のParquet対応では、データ本体を読まずにschema/metadataだけを読む
  - `time + 1ch` の列projection確認へ進む前に、列名・型・row group数を安定して取れることを確認する
  - 大データでもmetadata読込だけなら軽く確認できる
- `--schema` CLIモードを用意した
  - GUI操作なしで検出結果を再現できる
  - 今後、読み込み時間やメモリ測定のCLI入口にも拡張しやすい
- 選択中1ch読み込みでは、全列読み込みではなくParquet projectionを使う
  - Phase 1の目的である「必要列だけを読み、大容量で落ちない」検証に直結するため
  - 今回の大データ確認では `time + iu` の2列だけで約114 MiB保持になった
- 値配列は `f32` に寄せる
  - Python版でも波形列はfloat32化している
  - 描画用データとしては `f64` よりメモリ効率を優先する
  - `time` は検索・軸計算の基準として `f64` を維持する
- 最初のenvelopeは、表示範囲連動ではなくfull-range固定bucketで実装した
  - まず「全点を描画に渡さない」状態を作るため
  - 次段階でpan/zoom範囲に応じて再抽出する
- 4,096 bucketを暫定値にした
  - draw pointsは約8,000点で、画面幅オーダーに近い
  - 大データ10M点でもenvelope生成が約0.08 sに収まったため、Phase 1の初期値として妥当
- envelopeのbucket時刻は先頭・末尾時刻の中点にした
  - 等時間間隔前提のデータ仕様に合う
  - 今後のrange-aware LODでもbucket代表時刻として扱いやすい

## 詰まったこと・解決策

- `pyarrow` がsandbox内で `sysctlbyname` のCPU情報取得警告を出した
  - Parquet読み込み自体は成功したため、Phase 0では問題なしとして扱った
- 中・大データの合成ファイルは生成済み
  - 実機ログを使う場合は、機密扱いと匿名化方針を別途確認する
- 作図ロード後のチャンネル分散表示は既知リスクとして記録
  - Rust版ではロード時のUI状態更新と波形描画を分離し、Dash callback競合と同種の問題を避ける
- B-11 作図設定ロードは、リロード直後に自動クリックがDash側イベントとして拾われないケースがあった
  - 通常操作可能な状態では約0.63 sでロード完了
  - ベンチ結果では「処理時間」と「初期化直後の操作安定性」を分けて記録
- B-07 ズームリセットは当初の自動測定では戻らないように見えたが、手動再確認では再現しなかった
  - ズーム後の最終行X軸目盛り `0.2, 0.3, 0.4` が、全体表示時の `0, 0.2, 0.4, 0.6, 0.8` に戻ることを確認
  - 現時点では現行Dash版の既知リスクにはしない
- sandbox内では `cargo add eframe` / `cargo check` の初回依存取得でDNS解決に失敗した
  - 承認付きでcrates.ioへアクセスし、依存取得後は通常の `cargo check` が通る状態になった
- `eframe 0.34.1` のAPIが古いサンプルと異なり、`App::update` 実装では `ui` 未実装エラーになった
  - ローカルに取得されたcrateソースを確認し、`App::ui` と `Panel::show_inside` ベースへ修正した
- `cargo test` と `cargo check` を同時に走らせたため、一時的にbuild directory lock待ちが発生した
  - テスト完了後にcheckも正常終了。今後は重い初回ビルドでは直列実行を基本にする
- Arrowの配列変換で、最初は `ArrayAccessor` をジェネリック境界に置いたが、`PrimitiveArray<T>` ではなく参照型側に実装されていたためコンパイルエラーになった
  - 具象Arrow配列へdowncastするmacroに切り替えた
  - `Float64`, `Float32`, signed/unsigned integerを `f64` または `f32` に変換する形で整理
- `time + 1ch` 読み込み後、当初は `WaveformData::value_range()` を持たせたが、envelope側のrangeを使うようになって未使用警告が出た
  - range計算は `MinMaxEnvelope` 側に寄せ、未使用関数を削除した
- 小データで 4,096 bucket requested に対して 4,000 buckets built になった
  - 100,000点 / 4,096 bucket からbucket sizeが25になり、結果として 4,000 bucketになる
  - 実装上は問題なし。requested数は上限目標、built数が実描画点数の基準
